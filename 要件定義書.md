# 請求書リマインダー Slack ボット 要件定義書

**文書バージョン**: 1.1
**作成日**: 2026年3月3日
**ステータス**: 完成

---

## 1. プロジェクト概要

### 1.1 目的

請求書の送付漏れを防ぐため、Slack 上で相手先・期日を登録すると、期日の3日前から毎日リマインド通知を行う Slack ボットを構築する。担当者が完了チェックを行うまで通知が継続することで、対応漏れをゼロにすることを目指す。

### 1.2 背景・課題

- 請求書の送付期日を失念するリスクがある
- メモや口頭管理では抜け漏れが発生しやすい

---

## 2. システム構成

### 2.1 全体アーキテクチャ

```
[Slack] ←→ [Vercel (Next.js / API Routes)] ←→ [Upstash Redis]
                        ↑
               [cron-job.org]（毎朝9時に API を叩く）
```

| コンポーネント | 技術スタック |
|---|---|
| API | Next.js (App Router) on Vercel |
| Slack 連携 | @slack/web-api（公式 Node.js SDK） |
| データストア | Upstash Redis（無料枠） |
| スケジューラー | cron-job.org（無料・外部 Cron サービス） |
| 認証・環境変数管理 | Vercel Environment Variables |

## 3. 機能要件

### 3.1 ボット起動・登録フロー

#### 3.1.1 専用チャンネルのボタンによる起動

専用チャンネル（例: `#請求書リマインダー`）に Bot が常設メッセージを投稿・ピン留めする。
ユーザーはそのボタンを押すだけでフォームが開く。スラッシュコマンドは不要。

```
#請求書リマインダー チャンネル（ピン留め）

┌──────────────────────────────────────┐
│  📋 請求書リマインダー                │
│  送付期日が近い請求書を登録して       │
│  送り忘れを防ぎましょう               │
│                                      │
│  [ + 新しいリマインダーを登録 ]       │
└──────────────────────────────────────┘
```

| 項目 | 内容 |
|---|---|
| 起動方法 | 専用チャンネルのボタンをクリック |
| 実行場所 | 専用チャンネル（固定） |
| 応答 | モーダルダイアログを表示 |

#### 3.1.2 モーダル入力項目

| フィールド名 | 種別 | 必須 | 説明 |
|---|---|---|---|
| 送付先（相手先） | テキスト入力 | ○ | 例: 「株式会社〇〇」 |
| 送付期日 | 日付ピッカー | ○ | カレンダーから選択 |

#### 3.1.3 登録完了後の動作

- 登録確認メッセージを実行者の DM に送信
- データストアにリマインダー情報を保存

---

### 3.2 リマインド通知機能

#### 3.2.1 通知タイミング

| タイミング | 通知内容 |
|---|---|
| 期日の **3日前** | 初回リマインド通知を送信 |
| 期日の **2日前** | 2回目通知（未完了の場合） |
| 期日の **1日前** | 3回目通知（未完了の場合） |
| **期日当日** | 最終通知（未完了の場合） |
| 期日翌日以降 | 毎日継続（完了チェックまで） |

毎朝 cron-job.org が Vercel の API エンドポイントを呼び出し、対象リマインダーに通知を送る。

#### 3.2.2 通知メッセージ仕様

```
📋 請求書リマインダー

送付先  : 株式会社〇〇
期日    : 2026年3月10日（残り3日）

[ ✅ 送付完了 ] ボタン
```

#### 3.2.3 通知先

- 登録した本人への **DM**

---

### 3.3 完了チェック機能

- 通知メッセージ内の **「✅ 送付完了」ボタン**をクリックで完了
- 完了後はリマインド通知を停止
- データストアのステータスを「完了」に更新

## 4. 非機能要件

### 4.1 ホスティング・インフラ

| 項目 | 要件 |
|---|---|
| ホスティング | Vercel（Hobby プラン） |
| スケジューラー | cron-job.org（無料プラン・Vercel Pro 不要） |

### 4.2 スケジューラー（cron-job.org）

| 項目 | 内容 |
|---|---|
| 実行スケジュール | 毎日 午前 9:00 JST |
| 処理内容 | 通知対象リマインダーを全件走査し、条件を満たすものに通知送信 |
| 設定方法 | cron-job.org にアカウント作成 → Vercel の API URL と時刻を登録するだけ |

### 4.3 セキュリティ

| 項目 | 要件 |
|---|---|
| Slack 署名検証 | すべてのリクエストで `X-Slack-Signature` を検証 |
| Cron 認証 | cron-job.org からのリクエストに秘密トークンを付与して検証 |
| 環境変数管理 | Slack Token・Secret は Vercel 環境変数で管理 |

### 4.4 パフォーマンス

| 項目 | 要件 |
|---|---|
| Slack レスポンス | インタラクション受信から 3秒以内に応答（Slack 仕様上の制約）|

## 5. Slack アプリ設定要件

### 5.1 必要な OAuth スコープ

| スコープ | 用途 |
|---|---|
| `chat:write` | メッセージ送信・更新 |
| `im:write` | DM チャンネルを開く |
| `pins:write` | 専用チャンネルへの常設メッセージのピン留め |

### 5.2 設定項目

| 設定 | 内容 |
|---|---|
| Interactivity | ボタンクリック・モーダル送信の Request URL を設定 |
| Bot の招待 | 専用チャンネルに Bot をメンバーとして招待する |
| 初期セットアップ | デプロイ後に `/api/setup` を一度呼び出し、ボタンメッセージをチャンネルに投稿・ピン留め |

## 6. データ設計

### 6.1 リマインダーデータ構造（Upstash Redis に JSON で保存）

```json
{
  "id": "uuid-v4",
  "recipient": "株式会社〇〇",
  "due_date": "2026-03-10",
  "user_id": "U12345678",
  "status": "active",
  "created_at": "2026-03-03T10:00:00Z",
  "completed_at": null
}
```

キー設計:
- `reminder:{id}` → リマインダー本体
- `reminders:active` → アクティブな ID の集合（Redis Set）
