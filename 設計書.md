## 4. データ設計（Upstash Redis）

### 4.1 型定義

```typescript
// types/reminder.ts
export type Reminder = {
  id: string;           // UUID v4
  recipient: string;    // 送付先（例: "株式会社〇〇"）
  due_date: string;     // "YYYY-MM-DD"（JST 基準）
  user_id: string;      // 登録者の Slack User ID（例: "U12345678"）
  status: "active" | "completed";
  created_at: string;   // ISO 8601
  completed_at: string | null;
};
```

### 4.2 Redis キー設計

| キー | 型 | 内容 |
|---|---|---|
| `reminder:{id}` | String (JSON) | リマインダー本体 |
| `reminders:active` | Set | アクティブなリマインダー ID の集合 |

### 4.3 操作一覧

| 操作 | Redis コマンド |
|---|---|
| リマインダー保存 | `SET reminder:{id} {json}` + `SADD reminders:active {id}` |
| アクティブ一覧取得 | `SMEMBERS reminders:active` |
| リマインダー取得 | `GET reminder:{id}` |
| 完了処理 | `SET reminder:{id} {json（status更新）}` + `SREM reminders:active {id}` |

## 5. API エンドポイント設計

### 5.1 `POST /api/setup`

**役割**: デプロイ後に一度だけ呼ぶ初期セットアップ。専用チャンネルにボタンメッセージを投稿・ピン留めする。

**認証**: `Authorization: Bearer {CRON_SECRET}` ヘッダーを検証

**処理フロー**:
```
1. Authorization ヘッダー検証
2. chat.postMessage で REMINDER_CHANNEL_ID にボタンメッセージを投稿
3. pins.add で投稿したメッセージをピン留め
4. HTTP 200 を返す
```

**投稿するメッセージ（Block Kit）**:
```json
{
  "channel": "{REMINDER_CHANNEL_ID}",
  "blocks": [
    {
      "type": "section",
      "text": {
        "type": "mrkdwn",
        "text": "📋 *請求書リマインダー*\n送付期日が近い請求書を登録して、送り忘れを防ぎましょう。"
      }
    },
    {
      "type": "actions",
      "elements": [
        {
          "type": "button",
          "action_id": "open_registration_modal",
          "style": "primary",
          "text": { "type": "plain_text", "text": "+ 新しいリマインダーを登録" }
        }
      ]
    }
  ]
}
```

---

### 5.2 `POST /api/slack/interactions`

**役割**: すべての Slack インタラクション（ボタンクリック・モーダル送信）を受信する統合エンドポイント。

Slack はこのエンドポイントに `payload` という URL エンコードされた JSON を送ってくる。
`type` と `action_id` / `callback_id` を見て処理を分岐する。

---

#### ケース A: 登録ボタンクリック（`type: "block_actions"`, `action_id: "open_registration_modal"`）

専用チャンネルのボタンが押されたとき。

**処理フロー**:
```
1. 署名検証
2. payload をパース
3. trigger_id を取得
4. views.open でモーダルを表示（trigger_id は 3秒以内に使用する必要あり）
5. HTTP 200 を返す
```

**モーダル定義（Block Kit）**:
```json
{
  "type": "modal",
  "callback_id": "invoice_modal",
  "title": { "type": "plain_text", "text": "請求書リマインダー登録" },
  "submit": { "type": "plain_text", "text": "登録する" },
  "close": { "type": "plain_text", "text": "キャンセル" },
  "blocks": [
    {
      "type": "input",
      "block_id": "recipient_block",
      "label": { "type": "plain_text", "text": "送付先" },
      "element": {
        "type": "plain_text_input",
        "action_id": "recipient_input",
        "placeholder": { "type": "plain_text", "text": "例: 株式会社〇〇" }
      }
    },
    {
      "type": "input",
      "block_id": "due_date_block",
      "label": { "type": "plain_text", "text": "送付期日" },
      "element": {
        "type": "datepicker",
        "action_id": "due_date_input"
      }
    }
  ]
}
```

---

#### ケース B: モーダル送信（`type: "view_submission"`, `callback_id: "invoice_modal"`）

**処理フロー**:
```
1. 署名検証
2. payload をパース
3. recipient, due_date, user_id を抽出
4. UUID 生成 → Reminder オブジェクト構築
5. Redis に保存（SET + SADD）
6. conversations.open でユーザーの DM チャンネル ID を取得
7. chat.postMessage で登録完了 DM を送信
8. HTTP 200 を返す（body は空 → モーダルを閉じる）
```

**登録完了 DM メッセージ**:
```
✅ リマインダーを登録しました

送付先 : 株式会社〇〇
期日   : 2026年3月10日

期日の3日前からリマインドします。
```

---

#### ケース C: 完了ボタンクリック（`type: "block_actions"`, `action_id: "complete_reminder"`）

リマインダー通知 DM 内のボタンが押されたとき。

**処理フロー**:
```
1. 署名検証
2. payload をパース
3. action.value からリマインダー ID を取得
4. Redis から Reminder を取得（存在・status チェック）
5. status を "completed" に更新 → Redis 保存（SET + SREM）
6. chat.update でメッセージを完了表示に書き換え
7. HTTP 200 を返す
```

**完了後のメッセージ（chat.update で書き換え）**:
```
☑️ 送付完了済み

送付先 : 株式会社〇〇
期日   : 2026年3月10日
完了日 : 2026年3月8日
```

---

### 5.3 `POST /api/cron/notify`

**役割**: cron-job.org から毎朝呼び出され、通知対象リマインダーに DM 送信

**認証**: `Authorization: Bearer {CRON_SECRET}` ヘッダーを検証

**処理フロー**:
```
1. Authorization ヘッダーを検証（CRON_SECRET と一致しなければ 401）
2. SMEMBERS reminders:active → アクティブ ID 一覧取得
3. 各 ID について:
   a. GET reminder:{id} → Reminder 取得
   b. today（JST）と due_date の差分を計算
   c. 差分が 3日以下（due_date - today <= 3）なら通知対象
   d. conversations.open → DM チャンネル ID 取得
   e. chat.postMessage でリマインダー DM 送信
4. HTTP 200 を返す
```

**通知メッセージ（Block Kit）**:
```json
{
  "blocks": [
    {
      "type": "section",
      "text": {
        "type": "mrkdwn",
        "text": "📋 *請求書リマインダー*\n\n*送付先*: 株式会社〇〇\n*期日*: 2026年3月10日（残り3日）"
      }
    },
    {
      "type": "actions",
      "elements": [
        {
          "type": "button",
          "action_id": "complete_reminder",
          "style": "primary",
          "text": { "type": "plain_text", "text": "✅ 送付完了" },
          "value": "{reminder_id}"
        }
      ]
    }
  ]
}
```

**日数表示のルール**:
| 差分（due_date − today） | 表示テキスト |
|---|---|
| 3 | 残り3日 |
| 2 | 残り2日 |
| 1 | 明日が期日です |
| 0 | 本日が期日です |
| -1 以下 | 期日を N 日過ぎています |

## 7. Slack 署名検証

Slack はすべてのリクエストに `X-Slack-Signature` と `X-Slack-Request-Timestamp` ヘッダーを付与する。

```typescript
// lib/slack.ts（概要）
import crypto from "crypto";

export function verifySlackSignature(
  rawBody: string,
  signature: string,
  timestamp: string
): boolean {
  // 5分以上前のリクエストは拒否（リプレイアタック対策）
  if (Math.abs(Date.now() / 1000 - Number(timestamp)) > 300) return false;

  const baseString = `v0:${timestamp}:${rawBody}`;
  const hmac = crypto
    .createHmac("sha256", process.env.SLACK_SIGNING_SECRET!)
    .update(baseString)
    .digest("hex");

  return `v0=${hmac}` === signature;
}
```

> **Note**: Next.js App Router は `request.text()` で生テキストとして読み取ってから
> 署名検証 → パースの順で処理する（パース後では署名検証できない）。

## 9. Slack アプリ設定

### 9.1 Bot 名の設定場所

| 設定場所 | 項目 | 内容 |
|---|---|---|
| Basic Information | **App Name** | アプリの正式名称（OAuth 画面・アプリディレクトリに表示） |
| App Home | **App Display Name** | Bot がメッセージを送るときに表示される名前 |
| App Home | **Default Bot Name** | `@メンション` 時の名前 |

3か所を同じ名前（例: `請求書リマインダー`）に揃えておくと混乱がない。

---

### 9.2 Workspace 管理者のみが実施できる作業

> 以下は **Slack ワークスペースの管理者権限** が必要。一般メンバーには実施不可。

| # | 作業 | 場所 |
|---|---|---|
| 1 | **アプリのワークスペースへのインストール** | api.slack.com → Install to Workspace |
| 2 | **アプリインストールの承認**（ワークスペースの設定で承認制になっている場合） | Slack 管理画面 → 承認キュー |

> ワークスペースによっては「メンバーによるアプリインストール」を許可している場合もあるが、
> デフォルトは管理者承認制のため、管理者に依頼が必要。

---

### 9.3 管理者権限が不要な作業（開発者が自分で実施）

| # | 作業 | 場所 |
|---|---|---|
| 1 | Slack App の新規作成 | api.slack.com/apps → Create New App |
| 2 | Bot 名・表示名の設定 | Basic Information / App Home |
| 3 | Interactivity の Request URL 設定 | Interactivity & Shortcuts |
| 4 | OAuth スコープの追加 | OAuth & Permissions → Bot Token Scopes |
| 5 | Bot Token・Signing Secret の取得 | Basic Information / OAuth & Permissions |
| 6 | 専用チャンネルの作成・Bot 招待 | Slack クライアント上で `/invite @Bot名` |
| 7 | チャンネル ID の確認 | チャンネル名を右クリック → チャンネル詳細の末尾 |

---

### 9.4 設定手順（順番通りに実施）

#### Step 1: Slack App を作成する（管理者権限不要）
1. [api.slack.com/apps](https://api.slack.com/apps) → **Create New App** → **From Scratch**
2. App Name に `請求書リマインダー`（任意）を入力 → ワークスペースを選択

#### Step 2: Bot 名・表示名を設定する（管理者権限不要）
1. **Basic Information** → App Name を確認
2. **App Home** → App Display Name / Default Bot Name を同じ名前に設定

#### Step 3: Interactivity を設定する（管理者権限不要）
1. **Interactivity & Shortcuts** → Interactivity を ON
2. Request URL: `https://{your-app}.vercel.app/api/slack/interactions`

#### Step 4: OAuth スコープを追加する（管理者権限不要）
1. **OAuth & Permissions** → Bot Token Scopes に以下を追加:

| スコープ | 用途 |
|---|---|
| `chat:write` | メッセージ送信・更新 |
| `im:write` | DM チャンネルを開く（conversations.open） |
| `pins:write` | 常設メッセージのピン留め |

#### Step 5: ワークスペースにインストールする（⚠️ 管理者権限が必要）
1. **OAuth & Permissions** → **Install to Workspace**
2. 承認画面で「許可する」
3. 発行された **Bot Token**（`xoxb-...`）をコピーして Vercel 環境変数 `SLACK_BOT_TOKEN` に設定
4. **Basic Information** → Signing Secret をコピーして `SLACK_SIGNING_SECRET` に設定

#### Step 6: 専用チャンネルを準備する（管理者権限不要）
1. Slack で専用チャンネル（例: `#請求書リマインダー`）を作成
2. チャンネルで `/invite @請求書リマインダー` を実行して Bot を招待
3. チャンネル名を右クリック → **チャンネル詳細を表示** → 末尾の `C` から始まる ID をコピー → `REMINDER_CHANNEL_ID` に設定

> スラッシュコマンドは使用しないため **Slash Commands の設定は不要**。

## 10. 開発・デプロイ手順

### 10.1 ローカル開発

```bash
# 1. 依存パッケージインストール
npm install

# 2. .env.local に環境変数を記入

# 3. Slack へのローカル公開（ngrok を使用）
npx ngrok http 3000
# → 発行された URL を Slack App の Interactivity Request URL に一時設定

# 4. 開発サーバー起動
npm run dev
```

### 10.2 Vercel デプロイ

```bash
# Vercel CLI でデプロイ
npx vercel --prod

# 環境変数を設定
vercel env add SLACK_BOT_TOKEN
vercel env add SLACK_SIGNING_SECRET
vercel env add UPSTASH_REDIS_REST_URL
vercel env add UPSTASH_REDIS_REST_TOKEN
vercel env add CRON_SECRET
vercel env add REMINDER_CHANNEL_ID
```

> **Note**: Vercel ダッシュボードの **Settings → Deployment Protection** で
> **Vercel Authentication を Disabled** にすること。
> 有効のままだと Slack・cron-job.org からのリクエストが弾かれる。

### 10.3 初期セットアップ（デプロイ後に一度だけ）

```bash
# 専用チャンネルにボタンメッセージを投稿・ピン留め
curl -X POST https://{your-app}.vercel.app/api/setup \
  -H "Authorization: Bearer {CRON_SECRET}"
```

## 11. エラーハンドリング方針

| 状況 | 対応 |
|---|---|
| 署名検証失敗 | 403 を返す |
| CRON_SECRET 不一致 | 401 を返す |
| Redis 接続エラー | 500 を返す・ログ出力 |
| Reminder が存在しない or 完了済み | 処理をスキップ・200 を返す（冪等性確保） |
| Slack API エラー | ログ出力・cron は次の ID の処理を継続 |
